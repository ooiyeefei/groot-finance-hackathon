# Implementation Plan: Capacitor Mobile App (iOS)

**Branch**: `001-capacitor-mobile-app` | **Date**: 2026-02-21 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-capacitor-mobile-app/spec.md`

## Summary

Wrap the existing FinanSEAL Next.js web app in a Capacitor native shell for iOS distribution. The Capacitor WebView loads the live Vercel-hosted app via `server.url` (remote URL approach — static export is not viable due to middleware, server actions, and i18n dependencies). Native plugins provide camera, push notifications, deep linking, status bar integration, and crash monitoring. Phase 1 targets iOS App Store only; Android is deferred.

## Technical Context

**Language/Version**: TypeScript 5.9.3 (web app), Swift (Capacitor iOS shell — auto-generated)
**Primary Dependencies**: @capacitor/core, @capacitor/ios, @capacitor/camera, @capacitor/push-notifications, @capacitor/app, @capacitor/browser, @capacitor/status-bar, @capacitor/splash-screen, @sentry/capacitor@^2.4.1
**Storage**: Convex (existing, no new tables except `push_subscriptions` and `app_versions`), AWS SSM Parameter Store (APNs key)
**Testing**: Manual testing via TestFlight (iOS simulator + physical devices), existing web test suite for regression
**Target Platform**: iOS 16+ (Capacitor WebView pointing to Vercel-hosted Next.js app)
**Project Type**: Mobile (Capacitor shell wrapping existing web app)
**Performance Goals**: App launch < 3 seconds, push notification delivery < 30 seconds, crash rate < 1% of sessions
**Constraints**: Requires internet for initial load, Apple App Store review compliance (guideline 4.2), Clerk OAuth requires system browser (not WebView)
**Scale/Scope**: < 1000 users (Phase 1), single platform (iOS), 5 native plugin integrations

## Implementation Phases

### Phase A: Core Shell Setup (FR-001, FR-002, FR-007, FR-008, FR-010, FR-011, FR-012, FR-013)

**Goal**: FinanSEAL loads and functions correctly in the Capacitor iOS WebView.

1. Install Capacitor core packages (`@capacitor/core`, `@capacitor/cli`, `@capacitor/ios`)
2. Create `capacitor.config.ts` with remote URL pointing to Vercel
3. Run `npx cap add ios` to generate the Xcode project
4. Create `src/lib/capacitor/platform.ts` — platform detection utilities
5. Configure `allowNavigation` for Convex, Clerk domains
6. Enable `CapacitorCookies` for Clerk session persistence
7. Install `@capacitor/status-bar` and `@capacitor/splash-screen`
8. Configure splash screen and app icons in Xcode
9. Modify `src/components/ui/pwa-install-prompt.tsx` — suppress in Capacitor context
10. Modify `src/lib/hooks/use-pwa-install.ts` — detect Capacitor, skip PWA install logic
11. Test full app flow on iOS Simulator: login, navigation, all major features
12. Verify Convex real-time subscriptions work in WebView
13. Verify existing offline indicator and service worker behavior

**Validation**: All acceptance scenarios from User Story 1 pass on iOS Simulator.

### Phase B: OAuth Bridge (FR-002 — OAuth flows)

**Goal**: Clerk OAuth (Google, GitHub, etc.) works in Capacitor by routing through system browser.

1. Install `@capacitor/browser` and `@capacitor/app`
2. Create `src/lib/capacitor/auth-bridge.ts`:
   - Detect OAuth sign-in attempts in Capacitor context
   - Open OAuth URL via `Browser.open()` (uses `SFSafariViewController`)
   - Register `finanseal://` custom URL scheme in `Info.plist`
   - Listen for `appUrlOpen` callback event
   - Parse auth tokens and call Clerk `setActive()`
3. Test Google OAuth flow end-to-end on physical device
4. Test email/password flow (should work directly in WebView)
5. Verify session persistence across app cold starts

**Validation**: User can sign in with Google OAuth and email/password on iOS device.

### Phase C: Native Camera (FR-004)

**Goal**: Receipt capture uses the native camera with full device resolution.

1. Install `@capacitor/camera`
2. Create `src/lib/capacitor/camera-bridge.ts`:
   - Export a `capturePhoto()` function that detects platform
   - On native: use `Camera.getPhoto()` with full quality settings
   - On web: delegate to existing `getUserMedia()` browser flow
3. Modify `src/domains/expense-claims/components/mobile-camera-capture.tsx`:
   - Import camera bridge
   - Conditionally use native camera when `Capacitor.isNativePlatform()`
   - Preserve all existing browser camera functionality for web
4. Add `NSCameraUsageDescription` to `Info.plist`
5. Handle camera permission denial (clear message + settings redirect)
6. Test receipt capture on physical device, verify full resolution

**Validation**: All acceptance scenarios from User Story 2 pass. Web camera still works.

### Phase D: Push Notifications (FR-005)

**Goal**: Managers receive push notifications for pending expense approvals on iOS.

1. **Backend — Convex schema**:
   - Add `push_subscriptions` table to `convex/schema.ts`
   - Add `app_versions` table to `convex/schema.ts`
   - Run `npx convex deploy --yes`

2. **Backend — Convex functions**:
   - Create `convex/functions/pushSubscriptions.ts` (register, unregister, getByUserId)
   - Create `convex/functions/appVersions.ts` (getAppVersion query)
   - Modify `convex/functions/notifications.ts` — after creating an approval notification, schedule `sendPushNotification` internalAction

3. **Backend — APNs infrastructure**:
   - Create `infra/lib/apns-stack.ts` (CDK) — SSM parameters for APNs key, key ID, team ID
   - Deploy CDK: `cd infra && npx cdk deploy --profile groot-finanseal --region us-west-2`

4. **Backend — API route**:
   - Create `/api/v1/notifications/send-push/route.ts`
   - Implement APNs HTTP/2 sender (JWT signing with P8 key, direct HTTP/2 POST)
   - Handle APNs error codes (410 Unregistered → deactivate subscription)

5. **Client — Capacitor**:
   - Install `@capacitor/push-notifications`
   - Create `src/lib/capacitor/push-notifications.ts`
   - Request notification permission on first app launch
   - Register device token with Convex `pushSubscriptions.register`
   - Handle notification tap → deep link to approval item

6. **Apple Developer setup**:
   - Create APNs authentication key (P8) in Apple Developer Portal
   - Store P8 key, Key ID, Team ID in AWS SSM Parameter Store
   - Enable Push Notifications capability in Xcode project

7. Test full flow: submit expense → approver receives push → tap opens approval

**Validation**: All acceptance scenarios from User Story 3 pass.

### Phase E: Deep Linking (FR-006)

**Goal**: External URLs (from email, notifications) open directly to in-app content.

1. Configure Associated Domains capability in Xcode (`applinks:app.finanseal.com`)
2. Create `apple-app-site-association` file on Vercel (`.well-known/apple-app-site-association`)
3. Create deep link handler in `src/lib/capacitor/push-notifications.ts` (extend existing listeners):
   - Parse incoming URL path
   - Navigate to corresponding Next.js route in WebView
4. Test: tap link → app opens to correct content
5. Test: link without app installed → opens in Safari (fallback)

**Validation**: All acceptance scenarios from User Story 5 pass.

### Phase F: Update Mechanism & Crash Monitoring (FR-014, FR-015)

**Goal**: Force/soft update prompts and native crash reporting.

1. **Update mechanism**:
   - Create `src/lib/capacitor/update-checker.ts` — checks `appVersions.getAppVersion` on app launch
   - Create `src/components/ui/app-update-prompt.tsx` — force update (blocking) and soft update (dismissible banner)
   - Compare current native app version against `minimumVersion` and `latestVersion`

2. **Crash monitoring**:
   - Install `@sentry/capacitor@^2.4.1`
   - Create `src/lib/capacitor/sentry-init.ts` — conditional init (Capacitor vs web)
   - Create separate Sentry project `finanseal-mobile` (same org)
   - Add Xcode build phase for dSYM upload
   - Test crash reporting: trigger a JS error in the app, verify it appears in Sentry

**Validation**: Force update blocks app when below minimum version. Crashes appear in Sentry mobile project.

### Phase G: App Store Submission (FR-009)

**Goal**: FinanSEAL published on the Apple App Store.

1. **App Store Connect setup**:
   - Create app in App Store Connect with bundle ID `com.hellogroot.finanseal`
   - Configure app metadata: name, description, keywords, category (Finance)
   - Upload app icons (1024x1024 App Store icon)
   - Prepare screenshots for required device sizes (6.7", 6.1", etc.)
   - Write privacy policy and link it in App Store Connect
   - Configure App Privacy labels (data collection disclosures)

2. **TestFlight beta**:
   - Archive app in Xcode → Upload to App Store Connect
   - Distribute to internal team via TestFlight
   - Run through all acceptance scenarios on physical devices
   - Fix any issues found during beta testing

3. **App Store review submission**:
   - Fill in review information (demo account credentials, notes for reviewer)
   - Submit for review
   - Address any rejection feedback and resubmit if needed

**Validation**: App approved and available in App Store. SC-005 met.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Apple rejects app as "web wrapper" (guideline 4.2) | Medium | High | Include native camera, push, deep links, crash monitoring as native value-adds |
| Clerk OAuth breaks in WebView | High | High | Use @capacitor/browser for OAuth; email/password works as fallback |
| Convex WebSocket drops on background/foreground | Low | Medium | Convex has built-in reconnection; test thoroughly |
| Service worker conflicts with Capacitor WebView | Low | Medium | Test early in Phase A; disable SW in Capacitor if issues arise |
| APNs delivery failures | Low | Medium | Implement retry with exponential backoff; handle 410/400 token errors |
| Session cookie expiry (7-day ITP cap) | Medium | Medium | Test session persistence; implement token refresh if needed |
