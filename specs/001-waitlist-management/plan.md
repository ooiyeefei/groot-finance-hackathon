# Implementation Plan: Waitlist Management

**Branch**: `001-waitlist-management` | **Date**: 2026-02-22 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/001-waitlist-management/spec.md`

---

## Summary

Build a restaurant waitlist system allowing customers to join a queue for fully booked time slots, receive email notifications with time-limited claim links when a slot opens, and self-manage their entry via MAC-secured links. Staff manage the waitlist queue (view, reorder, notify, convert) from the admin dashboard. The feature spans two existing repositories sharing one Supabase PostgreSQL database, and introduces cron-based queue automation to groot-admin (which has no existing cron infrastructure).

---

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 20.x (both repos)
**Primary Dependencies**:
  - groot-admin: Next.js 15, Supabase JS client, Clerk 6.x, Resend, React Email, nanoid
  - groot-reservation: Next.js 15, Prisma 6.x, Resend, React Email, HMAC (node:crypto), nanoid

**Storage**: PostgreSQL via Supabase (shared instance). One new table: `waitlist_entries`. Schema change via Supabase migration in groot-admin; Prisma schema updated via `prisma db pull` in groot-reservation.

**Testing**: Integration tests in `groot-admin/tests/integration/` (existing pattern); no test framework currently in groot-reservation.

**Target Platform**: Web — Vercel (two separate deployments: admin app, customer app)

**Performance Goals**:
  - SC-001: Waitlist join to confirmation email < 60 seconds
  - SC-002: Cancellation to notification email < 5 minutes
  - SC-003: Admin waitlist view loads in < 30 seconds

**Constraints**:
  - No cron infrastructure exists in groot-admin — must be created with Vercel Cron
  - MAC auth pattern (HMAC-SHA256) for all customer-facing secure links — no login required
  - No role-based access control — all authenticated staff have full waitlist management access
  - Deposit collected at slot claim time, not waitlist join time
  - SMART table allocation mode only (PRESET mode deferred)

**Scale/Scope**: Restaurant-scale. Expected: 10–100 waitlist entries per restaurant per day. No horizontal scaling concerns.

## Rollout Notes

- Migration is additive only (new table, no changes to existing tables) — zero risk to existing reservation flow
- The cancellation trigger in `reservation-service.ts` must be `try/catch` wrapped — waitlist failure must not break cancellation
- Cron jobs are new infrastructure in groot-admin; add `CRON_SECRET` to Vercel environment variables before deploying
- Prisma schema addition is backward-compatible (new model, no changes to existing models)
