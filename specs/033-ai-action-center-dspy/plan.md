# Implementation Plan: Self-Improving Action Center (DSPy Feedback Loops)

**Branch**: `033-ai-action-center-dspy` | **Date**: 2026-03-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/033-ai-action-center-dspy/spec.md`

## Summary

Add DSPy self-improving feedback loops to the Action Center. Users provide feedback when dismissing/actioning insights; corrections accumulate and train a unified relevance classifier that learns which insights each business finds useful vs. noise. The classifier acts as a post-filter — algorithms generate candidates as today, DSPy decides what to surface.

**Key research finding**: Only 1 of 7 algorithms (employee expense spike) benefits from per-algorithm DSPy. Instead, we build a **unified post-filter relevance classifier** that works across all algorithms, learning per-business preferences.

```
└─────────────────────────────────────────────────────────────────┘
```

## Implementation Phases

### Phase 1: Feedback Capture (P1 — foundational)

**Goal**: Capture user feedback on every insight interaction.

**Changes**:
1. Schema: Add `userFeedback` to `actionCenterInsights`, create `action_center_corrections` table
2. Mutation: Extend `updateStatus` to accept `feedbackText`, create correction record on every status change
3. UI: Add optional feedback textarea to InsightCard dismiss dialog

**Deliverables**: Users can dismiss/action insights with feedback; corrections table populates.

### Phase 2: Optimization Pipeline (P2 — learning loop)

**Goal**: Train DSPy relevance classifier from accumulated corrections.

**Changes**:
1. Convex: Create `actionCenterOptimization.ts` with readiness check, training data queries, optimization orchestration
2. Lambda: Create `action_center_relevance.py` DSPy module (BootstrapFewShot)
3. Lambda handler: Add action center optimization case to `handler.py`
4. Infrastructure: Add EventBridge rule for weekly trigger

**Deliverables**: Weekly optimization runs, model versions created, quality gate enforced.

### Phase 3: Inference Integration (P2 — closing the loop)

**Goal**: Use trained model to filter insights at generation time.

**Changes**:
1. Model loader: Extend for business-scoped lookup (`module + businessId`)
2. Post-filter: Add relevance classification step to `actionCenterJobs.ts` before inserting insights
3. Observability: Log suppressed insight counts

**Deliverables**: Businesses with trained models see fewer false positive insights.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Insufficient corrections to train | High (early) | Low | Readiness gate prevents premature training; system works without DSPy |
| Model suppresses important insights | Medium | High | Quality gate + first-run auto-pass + conservative threshold |
| Convex bandwidth from corrections table | Low | Medium | EventBridge-first; corrections table is small (<1KB per record) |
| DSPy Lambda timeout | Low | Low | Existing Lambda has 5min timeout; action center training data is small |
