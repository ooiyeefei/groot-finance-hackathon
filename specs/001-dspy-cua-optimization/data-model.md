## Evaluation Queries (Computed, Not Stored)

Per-merchant evaluation scorecards are computed by Convex queries aggregating einvoice_request_logs:

```
Evaluation Report (computed from einvoice_request_logs)
├── merchantName
├── totalAttempts
├── successRate (%)
├── avgCostUsd
├── avgDurationMs
├── tierDistribution {tier1: N, tier2: N, tier2b: N}
├── hintEffectivenessRate (%)     # % of hints where next attempt succeeded
├── failureCategoryBreakdown {connectivity: N, form_validation: N, ...}
├── confidenceGateAccuracy (%)    # % correct predictions
└── trend (improving | stable | degrading)
```

## State Transitions

### Hint Effectiveness Lifecycle

```
1. Form fill fails → Troubleshooter generates cuaHint
   Log entry: generatedHint = "Click Company tab...", hintEffectivenessOutcome = "pending"

2. Next attempt for same merchant runs
   Previous log entry updated: hintEffectivenessOutcome = "helped" (if success) or "not_helped" (if fail)

3. MIPROv2 optimizer queries all entries with hintEffectivenessOutcome != "pending"
   Uses as training data to optimize troubleshooter prompts
```

### Optimization Pipeline Lifecycle

```
1. EventBridge triggers optimizer Lambda (every 3 days)
2. Optimizer queries Convex for recent request logs (training data)
3. Runs MIPROv2 on troubleshooter + BootstrapFewShot on recon
4. Evaluates optimized vs baseline using dspy.Evaluate
5. If optimized score > baseline: upload to S3 as latest.json
6. If optimized score <= baseline: retain baseline, log warning
7. Form fill Lambda picks up new module on next cold start
```
