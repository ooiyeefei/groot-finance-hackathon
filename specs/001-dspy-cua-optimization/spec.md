## User Scenarios & Testing *(mandatory)*

### User Story 1 - Self-Optimizing Troubleshooter (Priority: P0)

When a CUA form fill attempt fails, the troubleshooter diagnoses the failure and generates cuaHints (remediation instructions). Currently, these hints are generated by a basic one-shot prompt that never improves. With MIPROv2 optimization, the troubleshooter's prompts are automatically rewritten to maximize the probability that generated hints lead to success on the next attempt. The system tracks whether each hint actually helped, creating a closed feedback loop.

**Why this priority**: The troubleshooter is the core of the self-learning loop. Every other feature depends on having high-quality failure diagnosis. If hints are wrong, the system learns bad behaviors. MIPROv2 transforms the troubleshooter from "guessing" to "data-driven prompt optimization."

**Independent Test**: Submit an e-invoice request for a merchant that previously failed. Verify the troubleshooter generates a cuaHint. On the next attempt for the same merchant, verify the system uses the hint and succeeds. After 10+ attempts across merchants, verify that hint effectiveness rate exceeds 60% (vs current ~40% baseline).

**Acceptance Scenarios**:

1. **Given** a failed CUA form fill with error "validation_error: company name field empty", **When** the troubleshooter runs, **Then** it generates a specific cuaHint (e.g., "Click Company tab before filling company name field") AND the hint is saved to the merchant configuration
2. **Given** a merchant with a cuaHint from a previous failure, **When** a new CUA attempt runs for the same merchant, **Then** the CUA instructions include the learned hint AND the attempt succeeds
3. **Given** 20+ historical failure-then-success pairs in request logs, **When** the MIPROv2 optimizer runs offline, **Then** it produces an optimized troubleshooter module that generates more effective hints (measured by next-attempt success rate)
4. **Given** an optimized troubleshooter module, **When** it is serialized and cached, **Then** production loads the cached module without running the optimizer (zero latency overhead)

### User Story 2 - Cross-Merchant Recon Intelligence (Priority: P1)

Before CUA fills a form, a "recon" step describes the merchant's form layout using vision analysis. Currently, this description is generated fresh each time and discarded after use. With BootstrapFewShot, the system learns from successful recon-to-instruction patterns across merchants. When encountering a new merchant, the recon module includes proven examples from similar successful submissions (e.g., FamilyMart's react-phone-input pattern helps when encountering other merchants with similar phone widgets).

**Why this priority**: Recon quality directly determines CUA success. Poor recon = CUA wastes turns exploring. Good recon = CUA fills correctly on first pass. Cross-merchant intelligence means new merchants benefit from accumulated knowledge.

**Independent Test**: Submit an e-invoice request for a never-before-seen merchant. Verify the recon instructions include few-shot examples from similar successful merchants. Compare CUA turn count (actions taken) with and without bootstrapped recon.

**Acceptance Scenarios**:

1. **Given** 5+ successful form fill submissions with saved recon descriptions, **When** BootstrapFewShot runs on this training data, **Then** it selects the 3-4 most representative recon-to-instruction examples as few-shot demos
2. **Given** a new merchant form that resembles a known merchant's layout (react-phone-input, company toggle), **When** recon runs, **Then** the recon output includes a bootstrapped example showing how the similar form was successfully handled
3. **Given** bootstrapped recon examples are available, **When** CUA fills a new merchant's form, **Then** the average number of CUA turns decreases by at least 20% compared to cold-start (no examples)

### User Story 3 - Performance Measurement Dashboard (Priority: P1)

The system tracks cost and success/failure per attempt but lacks aggregated metrics. With the evaluation framework, operators can measure per-merchant success rates, cost efficiency, tier distribution, and hint effectiveness. This enables data-driven decisions about which merchants need optimization, which tiers are cost-effective, and whether the self-learning loop is actually improving.

**Why this priority**: You cannot improve what you cannot measure. Without metrics, optimization efforts are blind. The evaluation framework is the foundation for validating that all other features actually work.

**Independent Test**: Run the evaluation pipeline against the last 30 days of request logs. Verify it produces a per-merchant scorecard showing success rate, average cost, tier distribution, and hint effectiveness.

**Acceptance Scenarios**:

1. **Given** 30+ entries in request logs across 5+ merchants, **When** the evaluation pipeline runs, **Then** it produces a per-merchant report showing: success rate (%), average cost ($), tier distribution (% Tier 1 vs Tier 2 vs Tier 2B), average duration (seconds)
2. **Given** a merchant with cuaHints that were generated by the troubleshooter, **When** the evaluation measures hint effectiveness, **Then** it reports the percentage of cases where the hint led to success on the next attempt (feedback loop validation)
3. **Given** the evaluation framework produces a score for the current pipeline, **When** optimization runs and produces a new module, **Then** the evaluation score for the optimized module is higher than the baseline (quantitative proof of improvement)
4. **Given** a merchant's success rate drops below 50%, **When** the evaluation runs, **Then** it flags the merchant for manual review with specific failure category breakdown (connectivity vs form validation vs session issue)

### User Story 4 - Self-Healing CUA Instructions (Priority: P2)

CUA instructions sometimes miss required buyer fields or use vague descriptions instead of specific selectors. With hard constraints and soft preferences, the instruction generation module enforces that all required fields are addressed and prefers specific selectors when available. When a constraint fails, the system backtracks and regenerates instructions with error feedback, up to 3 retries.

**Why this priority**: Instruction quality determines CUA success. Constraint enforcement prevents the most common failure mode (missing required fields) without adding latency in the success path. Only triggers retries when instructions are genuinely deficient.

**Independent Test**: Generate CUA instructions for a merchant with known form configuration. Verify instructions include all required buyer fields (email, company name, TIN). Deliberately remove a required field from buyer_details and verify the constraint triggers a backtrack retry that adds the missing field.

**Acceptance Scenarios**:

1. **Given** buyer_details with email, company name, and TIN, **When** CUA instructions are generated, **Then** all three fields appear in the instructions (constraint passes on first attempt)
2. **Given** a merchant with saved form configuration containing selectors, **When** CUA instructions are generated, **Then** instructions reference the specific selectors (soft preference: prefer specific selectors over generic descriptions)
3. **Given** an instruction generation that omits the email field, **When** the constraint detects the omission, **Then** the system backtracks to the generation module, adds the error message to context, and retries (up to 3 times)
4. **Given** all 3 retry attempts fail to include a required field, **When** retries are exhausted, **Then** the system logs the failure and falls back to the current non-optimized instruction generation

### User Story 5 - Smart Tier 1 Skip (Priority: P2)

Tier 1 uses saved selectors from previous successes to fill forms quickly (~5s). But when a merchant updates their form, the saved selectors become stale and Tier 1 fails, wasting 5 seconds and potentially leaving partial fills that confuse Tier 2. With a confidence gate, the system predicts whether Tier 1 will succeed by comparing saved selectors against the current page structure. If confidence is below threshold, Tier 1 is skipped entirely.

**Why this priority**: Avoids wasted time and prevents partial-fill contamination. A failed Tier 1 is worse than skipping it because partial fills (some fields filled, some not) confuse CUA in Tier 2.

**Independent Test**: Load a merchant form where the saved selectors no longer match (simulated form update). Verify the confidence gate predicts failure and skips Tier 1. Verify the form fill proceeds directly to Tier 2 without partial fills.

**Acceptance Scenarios**:

1. **Given** a merchant with saved selectors that match the current form, **When** the confidence gate evaluates, **Then** confidence is above threshold and Tier 1 proceeds normally
2. **Given** a merchant whose form has changed (selectors no longer match), **When** the confidence gate evaluates, **Then** confidence is below threshold and Tier 1 is skipped with a log message indicating the skip reason
3. **Given** Tier 1 is skipped, **When** the form fill proceeds to Tier 2, **Then** the form is in a clean state (no partial fills from a failed Tier 1 attempt)
4. **Given** 10+ Tier 1 attempts tracked in logs, **When** the confidence gate is evaluated retrospectively, **Then** it correctly predicts >80% of Tier 1 outcomes

### User Story 6 - Intelligent Buyer Profile Selection (Priority: P3)

Account-gated merchants (e.g., 7-Eleven) require selecting a buyer profile from a list of saved profiles. Currently, this uses exact TIN matching which fails on name variations ("Groot Finance" vs "Groot Finance Sdn Bhd") and cannot handle multiple matches. With multi-step reasoning, the system performs: exact TIN match, then fuzzy name match, then recency-based selection if ambiguous.

**Why this priority**: Only affects account-gated merchants (currently 7-Eleven and similar). Lower volume but high per-case value since manual profile selection adds 2+ minutes per submission.

**Independent Test**: Present a list of 3 buyer profiles where TIN matches but names vary slightly. Verify the matching module selects the correct profile with step-by-step reasoning. Verify it recommends creating a new profile when no match is close enough.

**Acceptance Scenarios**:

1. **Given** a buyer profile list containing "Groot Finance Sdn Bhd" and buyer TIN matches, **When** the matcher runs with buyerName "Groot Finance", **Then** it selects "Groot Finance Sdn Bhd" with reasoning "TIN exact match + fuzzy name match (high similarity)"
2. **Given** multiple profiles with the same TIN but different branch names, **When** the matcher runs, **Then** it selects the most recently used profile with reasoning "Multiple TIN matches - selected most recent"
3. **Given** no profiles match the buyer TIN, **When** the matcher runs, **Then** it returns "create_new" with reasoning "No TIN match found - create new buyer profile"
4. **Given** a profile list with a TIN mismatch but very similar name, **When** the matcher runs, **Then** it does NOT select that profile (TIN is authoritative, name alone is insufficient)

### Edge Cases

- What happens when request logs have fewer than 10 entries (insufficient training data for optimization)?
  - System falls back to the current non-optimized modules and logs "Insufficient training data (N entries, need 10+)"
- What happens when the optimizer produces a module that performs worse than baseline?
  - Evaluation framework compares optimized vs baseline scores; if optimized is worse, the baseline is retained and a warning is logged
- What happens when vision analysis fails to provide form layout information in the recon step?
  - System falls back to current recon behavior (no bootstrapped examples) with a warning log
- How does the system handle merchant forms that change frequently (weekly form updates)?
  - The Tier 1 confidence gate detects stale selectors. The troubleshooter generates updated cuaHints. The evaluation framework tracks form-change frequency per merchant
- What happens when the optimized module file is corrupted or missing from the cache?
  - System falls back to the non-optimized module with a warning. The next offline optimization run regenerates the cached file
- What happens when a merchant has zero historical data (brand new merchant)?
  - BootstrapFewShot provides cross-merchant examples. All other features operate in default/fallback mode. After first success, the learning loop begins

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST run prompt optimization offline every 3 days via a scheduled trigger (not during production form fill) and serialize the optimized module to a file that can be loaded at runtime. The optimization frequency MUST be stored as a tuneable parameter
- **FR-002**: System MUST track hint effectiveness by correlating each cuaHint with the outcome of the next attempt for the same merchant (feedback loop)
- **FR-003**: System MUST store successful recon-to-instruction patterns as training examples, retaining the recon description, CUA instructions, and outcome
- **FR-004**: System MUST produce per-merchant evaluation reports by querying the extended request log table. Reports include: success rate, average cost, tier distribution, hint effectiveness rate, and failure category breakdown. The existing request log schema MUST be revamped to consolidate DSPy metrics (recon data, hint text, confidence scores, per-field results) rather than creating new tables
- **FR-005**: System MUST enforce hard constraints on CUA instruction generation: all required buyer fields (email, company name, TIN) must be present in generated instructions
- **FR-006**: System MUST support soft constraints that prefer specific selectors over generic field descriptions when form configuration is available
- **FR-007**: System MUST predict Tier 1 success probability by comparing saved selectors against current page structure, skipping Tier 1 when confidence is below threshold
- **FR-008**: System MUST use multi-step reasoning for buyer profile matching: exact TIN match, fuzzy name matching, recency-based disambiguation
- **FR-009**: System MUST fall back to non-optimized behavior when insufficient training data exists (fewer than 10 entries for optimizer, fewer than 5 successful submissions for few-shot learning)
- **FR-010**: System MUST log all optimization decisions (module loaded, fallback triggered, constraint retries, confidence scores) for observability
- **FR-011**: System MUST NOT increase production form fill latency by more than 500ms due to module loading and inference
- **FR-012**: System MUST compare optimized module performance against baseline before deploying; if optimized is worse, retain baseline
- **FR-013**: System MUST retain the existing 3-tier form fill architecture (Tier 1 saved selectors, Tier 2 CUA, Tier 3 troubleshooter) with enhancements layered on top

### Key Entities

- **Optimized Module Cache**: Versioned JSON files stored in S3 (`finanseal-bucket/dspy-modules/`) containing optimized prompts, few-shot examples, and configuration. One file per optimization target (troubleshooter, recon, instruction generator). Versioned with optimization timestamp and baseline score. Downloaded by form fill Lambda on cold start.
- **Training Dataset**: Curated set of examples extracted from request logs for optimizer training. Includes: merchant name, error message, generated hint, next-attempt outcome, recon description, CUA instructions, tier reached, form config snapshot.
- **Request Log (extended)**: The existing request log table is revamped to include DSPy-specific fields: recon description, generated cuaHint text, hint effectiveness outcome (did next attempt succeed?), confidence gate score, per-field fill results, and failure category classification. This consolidation avoids creating redundant tables while enabling evaluation queries directly against the source data.
- **Evaluation Report (computed)**: Per-merchant scorecard computed by aggregating request log data. Includes: success rate, cost metrics, tier distribution, hint effectiveness, failure categories. Generated on-demand via Convex query, not stored as a separate entity.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Troubleshooter hint effectiveness rate improves from ~40% baseline to 60%+ after optimization (measured as: % of cases where the generated hint led to success on the next attempt)
- **SC-002**: Average CUA turns (actions taken) for new merchants decreases by 20%+ when bootstrapped recon examples are available vs cold-start
- **SC-003**: Per-merchant evaluation reports are available on-demand, covering 100% of merchants with 10+ form fill attempts
- **SC-004**: CUA instruction generation never omits required buyer fields (email, company name, TIN) - 0% omission rate enforced by constraints
- **SC-005**: Tier 1 confidence gate correctly predicts Tier 1 outcomes in 80%+ of cases (true positive rate for skipping stale configs)
- **SC-006**: Buyer profile matching selects the correct profile in 90%+ of cases for account-gated merchants, reducing manual selection by 80%
- **SC-007**: Overall e-invoice form fill success rate improves from current baseline by at least 10 percentage points within 30 days of deployment
- **SC-008**: No production latency regression - form fill total time does not increase by more than 500ms on average due to enhancements
- **SC-009**: Cost per successful e-invoice submission decreases by 15%+ (fewer wasted CUA calls due to better confidence gating and better instructions)

## Assumptions

- The existing request logs table has sufficient historical data (20+ entries across 5+ merchants) to begin training. If not, the system operates in fallback mode until data accumulates.
- Gemini 3.1 Flash-Lite is used for all non-CUA inference (recon, troubleshooter, confidence gate, buyer matching). CUA model is only used for Tier 2 visual form filling.
- Optimized modules are stored as versioned JSON files in S3 (finanseal-bucket) under a `dspy-modules/` prefix. The form fill Lambda downloads the latest module on cold start (~200ms overhead). This decouples optimization from Lambda deployment — modules can be updated without redeploying the Lambda.
- The offline optimization pipeline runs every 3 days via EventBridge schedule. Frequency is a tuneable parameter (stored in SSM or Lambda environment) that can be adjusted based on evaluation data and data accumulation rate.
- Form configurations and cuaHints are stored per-merchant in the existing merchant configuration table, not per-business-tenant.
- The 0.7 confidence threshold for Tier 1 gating is a starting default that can be tuned based on evaluation data.
