# API Contracts: PDPA Data Subject Rights & Clerk/Convex Name Sync

**Date**: 2026-03-03

## P1: Name Sync

### POST /api/v1/users/update-clerk-profile

Updates a user's name in Clerk. Called by the existing profile PATCH endpoint when name changes are detected.

**Authentication**: Clerk session (authenticated user)
**Authorization**: Admin/owner for editing others; any authenticated user for editing self

**Request**:
```json
{
  "clerk_user_id": "user_2abc123...",
  "first_name": "Jonathan",
  "last_name": "Smith"
}
```

**Response (200)**:
```json
{
  "success": true,
  "clerk_user_id": "user_2abc123..."
}
```

**Error Responses**:

| Status | Code | Description |
|--------|------|-------------|
| 400 | `INVALID_NAME` | Name validation failed (empty, too short) |
| 401 | `UNAUTHORIZED` | No valid session |
| 403 | `FORBIDDEN` | Non-admin trying to edit another user |
| 404 | `USER_NOT_FOUND` | Clerk user ID not found or deactivated |
| 500 | `CLERK_API_ERROR` | Clerk API call failed |

**Error Response Body**:
```json
{
  "success": false,
  "error": "Failed to update profile: user not found in identity provider"
}
```

### PATCH /api/v1/users/profile (modified behavior)

**Existing endpoint** — modified to call Clerk `updateUser` when `full_name` changes.

**Current behavior**: Updates Convex only.
**New behavior**: When `full_name` is in the payload, calls `POST /api/v1/users/update-clerk-profile` first. If Clerk update succeeds, proceeds with Convex update. If Clerk update fails, returns error without touching Convex.

No request/response format changes.

## P3: Download My Data

### Convex Query: `exportJobs.getMyDataExport`

New Convex query that gathers a user's data across all businesses and modules.

**Authentication**: Clerk identity token (via Convex auth)

**Args**:
```typescript
{
  // No args — uses authenticated user's identity
}
```

**Returns**:
```typescript
{
  profile: {
    email: string
    fullName: string | null
    homeCurrency: string | null
    timezone: string | null
    language: string | null
    createdAt: string  // ISO date
  }
  businesses: Array<{
    businessId: string
    businessName: string
    role: string
    modules: {
      expense_claims: Array<Record<string, unknown>>
      invoices: Array<Record<string, unknown>>
      leave_requests: Array<Record<string, unknown>>
      accounting_entries: Array<Record<string, unknown>>
    }
  }>
}
```

**Error cases**:
- Unauthenticated → returns `null`
- No business memberships → returns profile only with empty `businesses` array

### Client-Side ZIP Structure

Generated by the React component using JSZip:

```
groot-finance-my-data-YYYY-MM-DD.zip
├── profile.csv
│   Columns: Email, Full Name, Currency, Timezone, Language, Account Created
│
├── {Business-Name}/
│   ├── expense_claims.csv
│   │   Columns: (from existing expense field definitions)
│   ├── invoices.csv
│   │   Columns: (from existing invoice field definitions)
│   ├── leave_requests.csv
│   │   Columns: (from existing leave field definitions)
│   └── accounting_entries.csv
│       Columns: (from existing accounting field definitions)
│
└── {Business-2-Name}/   (if multi-business)
    └── ...
```

**Rules**:
- Empty modules (0 records) are omitted from ZIP
- Business folder names are sanitized (remove special chars)
- CSV uses existing field definitions from `FIELD_DEFS` in `exportJobs.ts`
- Uses "Generic" prebuilt template field mappings for readable column headers
